# Changli OJ × MiaowTest 二改/爆改 PRD

更新时间：2026-03-01
状态：已启动（M0 已落地）

## 核心目标 (Mission)
基于开源项目 `MiaowTest`，在不推翻可用资产的前提下完成两阶段重构：
1. 二改（兼容式重构）：先把 Web First 主链路做稳并可上线。
2. 爆改（架构级重构）：把单体 API 演进为可持续多端、多角色、可付费的产品底座。

## 用户画像 (Persona)
1. 学生用户：网页端刷题、错题复盘、进度续学。
2. 运营/教研：题库导入、编辑、发布、质量校验。
3. 管理员：权限、审计、系统稳定性。

## V1: 最小可行产品 (MVP)
1. Web First 登录注册（邮箱/账号密码），不依赖微信登录。
2. 题库列表/题型浏览/答题接口稳定可用。
3. 题库导入链路可追踪（任务状态 + 失败明细）。
4. 进度与错题闭环。
5. 后台发布控制：未发布题库不可见。

## V2 及以后版本 (Future Releases)
1. 小程序端同源接入，统一权限模型。
2. AI 本题讲解（缓存优先，成本可控）。
3. 会员/题库包与支付体系。
4. 数据看板与推荐系统。

## 关键业务逻辑 (Business Rules)
1. 发布规则：仅 `isPublish=1` 的题库/题型对前台可见。
2. 判题规则：客观题自动判分，主观题先展示参考答案。
3. 错题规则：判错入错题本，重做正确后可移除。
4. 进度规则：每次提交更新用户进度与统计。
5. 权限规则：用户端/后台接口分层隔离，后台必须鉴权。

## 数据契约 (Data Contract)
沿用并标准化当前模型：
1. `consumers`：用户账号与画像。
2. `exams`：题库元数据、发布状态、认证要求。
3. `select/blank/judge/short`：题目实体。
4. `wrongQuestions`、`progress`：学习轨迹。
5. `importJobs`（新增）：导入任务及失败报告。

## 二改方案（兼容式重构，4-6 周）

### 目标
在不破坏现有 `uniappAPI` 的前提下，新增 `api/v1` 并完成 Web First 可上线能力。

### 改造范围
1. API 分层：补齐 `api/v1/auth|quizzes|progress`，旧路由继续可用。
2. 安全基线：密码哈希、登录限流、token 验证统一。
3. 数据兼容：老明文密码平滑升级为哈希，不强制迁移停机。
4. 前端接入：H5 优先走 `api/v1`，逐步替换 `uniappAPI` 直连。

### 验收标准
1. 新注册账号不再明文入库。
2. 老账号首次登录后自动升级为哈希。
3. `api/v1` 可独立支撑 Web 登录与题库浏览。
4. 原小程序接口不回归破坏。

## 爆改方案（架构级重构，8-12 周）

### 目标
从“接口集合”升级为“领域服务 + 任务系统 + 可观测”架构。

### 改造范围
1. 领域重组：auth / quiz / learning / import / admin 服务化。
2. 数据统一：题型分表逐步抽象为统一 `QuestionDTO` 读写层。
3. 导入管道：`sync -> clean -> validate -> import -> publish` 全链路。
4. 任务系统：BullMQ + Redis，导入与 AI 任务异步化。
5. 平台化能力：审计日志、告警、指标看板、Feature Flag。

### 风险与控制
1. 风险：旧前端耦合路径多。
   控制：API 双轨（旧接口不删，新增接口渐进替换）。
2. 风险：数据模型历史包袱。
   控制：先加中间层 DTO，再做存储层重构。
3. 风险：AI 与支付导致范围失控。
   控制：统一归入 V2+，V1 不并行开发。
